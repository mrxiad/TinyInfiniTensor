## 2. 代码修改概览

## 2.1 构造函数默认 permute 行为修正

文件：`src/operators/transpose.cc`

修改点：

1. 原实现在 `permute.empty()` 分支里直接写 `transposePermute[i]`，但没有 `resize`，有越界风险。
2. 现在先 `transposePermute.resize(rank)`。
3. 同时采用 ONNX 默认语义：`permute` 为空时，默认按维度倒序转置（例如 NCHW -> WHCN 的维度顺序反转）。

最终行为：

- `permute` 为空：默认 `[rank-1, ..., 0]`
- `permute` 非空：必须 `size == rank`，否则断言失败

## 2.2 `inferShape` 实现完成

文件：`src/operators/transpose.cc`

实现逻辑：

1. 读取输入维度 `input_dim` 与 `rank`。
2. 校验 `transposePermute.size() == rank`。
3. 校验每个轴：
   - `0 <= axis < rank`
   - 轴不重复（使用 `visited`）
4. 构造输出维度：
   - `output_dim[i] = input_dim[transposePermute[i]]`
5. 返回 `{{output_dim}}`。

这正是 transpose 的标准 shape 映射。

## 3. 算法说明

## 3.1 形状映射规则

设输入 shape 为：

- `input_dim = [d0, d1, d2, ...]`

permute 为：

- `p = [p0, p1, p2, ...]`

则输出 shape 为：

- `output_dim[i] = input_dim[p[i]]`

例如：

- 输入 `[1, 2, 3, 4]`
- `permute = [0, 2, 1, 3]`
- 输出 `[1, 3, 2, 4]`

## 3.2 校验策略

为了避免非法 transpose 配置带来的隐式错误，增加了两层保护：

1. 长度校验：`permute.size()` 必须等于 `rank`。
2. 置换合法性校验：
   - 每个轴都在合法范围内
   - 不能重复（必须是一个完整排列）

这样能尽早在构图阶段暴露问题，而不是把错误拖到 runtime。

## 4. 与框架流程的关系

`OperatorObj::checkValid` 在构图时会调用 `inferShape`：

1. `inferShape` 成功 -> 创建输出 Tensor
2. `inferShape` 失败（`nullopt`）-> 构图失败

因此作业2一旦未实现，`TransposeObj` 在图里创建就会失败，
连 `test_transpose` 都无法走到后续逻辑。

本次完成后，transpose 在以下两个层面恢复正常：

1. 图层：能创建正确 shape 的输出 Tensor
2. 运行层：`test_nativecpu_transpose` 能拿到正确布局输出
