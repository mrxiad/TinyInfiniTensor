# 作业1总结：Allocator 与图内存分配（详细版）

## 1. 作业目标

根据课程文档，作业1需要完成三件核心工作：

1. 在 `Allocator` 中设计并实现可复用的内存块管理结构。
2. 在 `Allocator::alloc/free` 中实现“模拟分配 + 回收 + 合并”。
3. 在 `GraphObj::dataMalloc` 中把图上的 Tensor 生命周期映射为内存复用策略，并最终给每个 Tensor 绑定可访问的数据指针。

对应文件：

- `include/core/allocator.h`
- `src/core/allocator.cc`
- `src/core/graph.cc`

## 2. 改动总览

### 2.1 `include/core/allocator.h`

新增空闲块容器：

- `std::map<size_t, size_t> freeBlocks;`
- 含义：`key=块起始偏移`，`value=块大小`

这样可以按地址有序管理空闲块，便于：

1. 查找可复用块
2. 与前后相邻块合并
3. 对尾部空闲块进行“回缩（trim）”

### 2.2 `src/core/allocator.cc`

实现了 `alloc` 与 `free` 的核心逻辑：

- `alloc`：优先复用空闲块，否则从 `used` 末尾扩展。
- `free`：
  - 若释放的是尾部块，直接回退 `used` 并持续回收与尾部相邻的空闲块。
  - 若释放的是中间块，插入 `freeBlocks`，再与前后邻接块合并；如果合并后又碰到尾部，再继续回退。

同时补了一个稳健性改动：

- `getAlignedSize(0)` 返回 0，避免 `(size - 1)` 的无符号下溢问题。

### 2.3 `src/core/graph.cc`

实现了 `GraphObj::dataMalloc`：

1. 先拓扑排序。
2. 基于图关系构建 `remainUses`（每个 Tensor 剩余被消费次数）。
3. 标记 `graphOutputs`（图输出）为常驻，不参与提前释放。
4. 在遍历算子时：
   - 确保输入/输出 Tensor 都已分配偏移。
   - 每消费一次输入，`remainUses--`。
   - 若某输入 `remainUses==0` 且不是图输出，则立即 `allocator.free(...)`，让后续 Tensor 复用该空间。
5. 最后 `allocator.getPtr()` 真实申请一整块内存，并将每个 Tensor 的偏移映射成实际指针，写入 `Blob`。

## 3. Allocator 设计与算法细节

### 3.1 内部状态语义

`Allocator` 内几个关键字段：

- `used`：当前“逻辑堆顶”偏移（不是累计申请量，而是当前末端位置）。
- `peak`：历史最大 `used`，用于最终一次性真实申请内存。
- `alignment`：对齐粒度（默认 8 字节）。
- `freeBlocks`：中间可复用空闲块（有序 map）。
- `ptr`：真实分配指针，调用 `getPtr()` 后不允许再做模拟分配/回收。

### 3.2 `alloc(size)` 具体流程

1. 断言 `ptr == nullptr`，保证还处于“规划阶段”。
2. 对齐 `size`。
3. 在 `freeBlocks` 里找第一个 `blockSize >= size` 的块（按地址遍历）。
4. 找到则复用：
   - 返回 `addr = blockStart`
   - 若块更大，剩余部分重新放回 `freeBlocks`
5. 若没找到：
   - `addr = used`
   - `used += size`
   - `peak = max(peak, used)`

关键点：

- 优先复用空闲块，减少峰值。
- `peak` 只在扩堆时增长。

### 3.3 `free(addr, size)` 具体流程

释放分两类：

### A. 尾部释放（`addr + size == used`）

这是最重要的路径：

1. `used = addr`，直接回缩。
2. 继续检查 `freeBlocks` 中是否存在“正好贴着新堆顶”的空闲块。
3. 若存在，继续回缩并删除该块（循环）。

这保证了“尾部连续空闲区”会尽量被回收，避免 `used` 虚高。

### B. 中间释放（非尾部）

1. 将块插入 `freeBlocks`（若 key 重复则断言失败，避免重复释放）。
2. 与前一个相邻块合并。
3. 与后一个相邻块连续合并。
4. 若合并后块恰好触达 `used`，再执行一次尾部回缩。

### 3.4 为什么这样能通过 `testAllocWithEndFreeBlock`

该测试核心是：

1. 先分配 `a,b,c`
2. 释放 `c`（尾部块）
3. 分配更大的 `d`
4. 期望 `offsetD == offsetC`

若只把 `c` 放入空闲表，不回缩 `used`，则 `d` 可能从更高地址分配，测试失败。

当前实现通过“尾部回缩”让 `used` 回到 `offsetC`，因此 `d` 可以从 `offsetC` 起分配，测试通过。

## 4. `GraphObj::dataMalloc` 的生命周期复用策略

### 4.1 问题本质

图执行时，Tensor 的生命周期并不重叠完全：

- 某些中间 Tensor 在“最后一次被消费后”即可回收。
- 图输出 Tensor 需要保留到执行完成后读取，不能提前回收。

因此 `dataMalloc` 做的是“静态内存规划”：

- 先算偏移
- 最后一次性申请 `peak` 大小的真实内存

### 4.2 关键数据结构

- `offsets[tensor]`：Tensor -> 分配偏移
- `bytes[tensor]`：Tensor -> 占用字节
- `remainUses[tensor]`：Tensor 剩余消费次数（初值为 `targets.size()`）
- `graphOutputs`：图输出集合（不提前 free）

### 4.3 执行顺序（按拓扑）

对每个 op：

1. 确保所有输入/输出 Tensor 已有偏移（`ensureAllocated`）。
2. 消费输入后 `remainUses--`。
3. 当输入 `remainUses==0` 且不在 `graphOutputs`，执行 `allocator.free`。

这个顺序的意义：

- 当前 op 仍需要输入数据，所以先不 free。
- op 执行完后，该输入的“本次消费”结束，再判断是否可回收。

### 4.4 最终绑定

1. `basePtr = allocator.getPtr()`
2. 每个 Tensor 使用 `basePtr + offsets[tensor]` 生成 `BlobObj`
3. `tensor->setDataBlob(...)`

这样所有 Tensor 都共享同一块大内存，但拥有不同偏移；被复用的 Tensor 会映射到同一偏移区间。

## 6. 复杂度分析

### 6.1 Allocator

- `alloc`：
  - 最坏需要线性扫描空闲块（当前实现按地址遍历）
  - 复杂度约 `O(F)`，`F` 为空闲块数量
- `free`：
  - 插入与邻接合并以 map 操作为主
  - 复杂度约 `O(log F)` + 常数次邻接检查

### 6.2 Graph dataMalloc

设：

- `T` = Tensor 数
- `E` = 图中输入边（按 op 输入计数）

复杂度约：

- 初始化计数：`O(T)`
- 遍历 op 输入输出并分配/释放：`O(E * alloc/free_cost)`
- 最终绑定：`O(T)`

## 7. 边界与健壮性处理

本次实现已处理：

1. 规划阶段保护：`alloc/free` 都断言 `ptr == nullptr`，防止“真实分配后再改规划”。
2. 对齐：所有分配/释放都统一按 `alignment` 对齐。
3. 重复 free 防护：同地址重复插入空闲块会断言失败。
4. 尾部连续回缩：避免 `used` 虚高。
5. `size==0` 的对齐保护：避免下溢。

## 8. 现阶段限制

1. `alloc` 采用“按地址首个可用块”策略，不是 best-fit；在极端碎片场景下不是最优。
2. `GraphObj::dataMalloc` 是静态规划，默认一次构图一次规划；未实现多轮动态重规划。
3. 目前项目其他作业未完成，导致不能直接用全量测试证明整仓功能完整。

## 9. 后续可优化方向（作业1完成后的工程增强）
